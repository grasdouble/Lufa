## Quick Start

### Working on a New Subject

1. **Create subject folder:**

   ```bash
   mkdir -p _bmad-output/subjects/{subject-name}/{analysis,planning,implementation}
   ```

2. **Add subject README:**

   ```bash
   touch _bmad-output/subjects/{subject-name}/README.md
   ```

3. **Follow naming conventions:**
   - See [NAMING-CONVENTIONS.md](./NAMING-CONVENTIONS.md)
   - Use subject-specific filenames
   - Prefix with phase if helpful

### Creating an ADR

1. **Find next number:**

   ```bash
   ls _bmad-output/adrs/ | grep ADR | tail -1  # Get last ADR number
   ```

2. **Create ADR:**

   ```bash
   touch _bmad-output/adrs/ADR-{next-number}-PROPOSED-{decision-name}.md
   ```

3. **Update index:**
   - Edit `_bmad-output/adrs/README.md`
   - Add entry to the table

### 2. **ADRs** (Architecture Decisions)

Architecture Decision Records are kept **separate** from subjects because:

- They're cross-cutting (can affect multiple subjects)
- They need global sequential numbering
- They're referenced across subjects

**Location:** `_bmad-output/adrs/`

**Format:** `ADR-{number}-{STATUS}-{short-name}.md`

**Example:**

```
ADR-001-IMPLEMENTED-modes-vs-themes-separation.md
ADR-002-IMPLEMENTED-html-attributes-naming.md
ADR-003-PROPOSED-badge-variant-strategy.md
```

**See:** [adrs/README.md](./adrs/README.md) for full guidelines

---

### 3. **Global** (Cross-Project Files)

Files that track state across all subjects:

- `bmm-workflow-status.yaml` - BMM workflow progress
- (Future: Sprint tracking, metrics, etc.)

**Location:** `_bmad-output/global/`

**Guideline:** Only files that are:

- Singleton (one per project)
- Cross-cutting (affect all subjects)
- State tracking (not decisions - those are ADRs)

---

## File Naming Conventions

### General Rules

1. **Use kebab-case:** `theme-integration`, not `ThemeIntegration`
2. **Be specific:** Include subject context where helpful
3. **Add dates for analysis:** `analysis-2026-01-26.md`
4. **No dates for specs:** They're living documents (git tracks history)

### Subject Files

Within a subject folder, names can be simpler (subject is in path):

```
subjects/theme-integration/
â”œâ”€â”€ README.md                           # Not "theme-integration-readme.md"
â”œâ”€â”€ analysis/
```

## FAQ

**Q: Can I have multiple subjects in progress?**  
A: Yes! Each subject is independent. Work on as many as needed.

**Q: Should I create a subject for small tasks?**  
A: No. Subjects are for significant features/projects. Small tasks can go directly in code with PR docs.

**Q: Where do sprint reports go?**  
A: In the relevant subject: `subjects/{name}/implementation/sprint-{n}-report.md`

**Q: Can I reorganize subject files?**  
A: Yes, within the subject folder. Just update references if needed.

**Q: What if an ADR affects multiple subjects?**  
A: Perfect! That's why ADRs are separate. Reference it from multiple subject READMEs.
